## 📘 What this is

open-ai-transformation-maturity-model is an open framework for checking how ready a software team is for AI change. It helps you look at your current setup, compare it with a target state, and see where you stand on the path from copilots to autonomous software factories.

It is built for software groups that want a clear view of AI use across delivery, engineering, governance, and day-to-day work. It uses plain checks, evidence, and simple scoring so you can assess progress without guesswork.

## 🧠 Maturity stages

A common path in this model looks like this:

### 1. Explorer

Teams try AI tools in small ways. Use is local, and people test what helps them most.

### 2. Assisted

AI helps with tasks like writing code, making tests, or summarizing work. People still make the main choices.

### 3. Integrated

AI tools fit into the normal delivery flow. Teams use them in planning, coding, testing, and review.

### 4. Managed

Teams set rules, metrics, and review steps. AI use is part of team standards and governance.

### 5. Autonomous

Teams use AI agents and automated flows for parts of software delivery, with strong controls and human review where needed

## 🧾 How to run an assessment

Use a short workshop with the team.

1. Choose one product or delivery stream.

2. Invite people from engineering, operations, and leadership.

3. Review each section of the model.

4. Ask for proof, not guesses.

5. Score each area based on current practice.

6. Record the reasons for each score.

7. Note the next step for each weak area.

8. Set a date for the next review.

This works best when people use real examples from recent work, such as pull requests, release notes, incident reviews, or platform reports.

## 🔐 Governance and safe use

AI use in software work needs clear control points. This model can help teams think about:

- Approved tools
- Data handling rules
- Access control
- Human review for high-risk work
- Audit notes for AI-assisted changes
- Model and prompt review
- Security checks
- Policy use in daily work

This helps teams move faster without losing control of quality or risk.

## 📊 Scoring tips

Keep scoring simple and steady.

- Use the same scale across all areas
- Base scores on proof
- Avoid score inflation
- Record why you chose each score
- Revisit scores after each improvement cycle

A simple scale might look like this:

- 1 = Early use
- 2 = Basic use
- 3 = Working practice
- 4 = Controlled and repeatable
- 5 = Broad and well managed
